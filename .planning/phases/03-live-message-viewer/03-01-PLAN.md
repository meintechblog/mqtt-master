       - Maintain a Map<socket, Set<string>> for per-client subscriptions
       - On client connect: add to map with empty Set
       - On client message (JSON parsed):
         - action "subscribe": validate topic is non-empty string, call app.mqttService.subscribe(topic), add topic to client's Set, send {"type":"subscribed","topic"} confirmation
         - action "unsubscribe": call app.mqttService.unsubscribe(topic), remove from client's Set, send {"type":"unsubscribed","topic"} confirmation
         - Invalid/missing action: send {"type":"error","message":"Unknown action"}
       - Listen to app.mqttService 'message' event: for each client, check if any topic in their subscription Set matches the message topic (use mqtt-match pattern matching or simple Set.has -- note: MQTT wildcards are handled by the broker, so Set.has on the subscribed pattern is NOT correct for filtering; instead, since the broker handles wildcard matching, forward ALL messages to clients that have ANY active subscription and let the broker's subscription filtering do the work. Actually the simplest correct approach: subscribe via MqttService which subscribes on the broker, and the broker only delivers matching messages. But since multiple clients share one MqttService connection, ALL subscribed messages arrive on the single 'message' event. So we need to track which topics each client subscribed to and use MQTT topic matching to determine if a message matches any of a client's subscriptions.)
       - **MQTT topic matching logic**: Install no new dependencies. Implement a small `mqttTopicMatch(pattern, topic)` function inline in ws-messages.js that handles `+` (single level) and `#` (multi-level) wildcards. This is ~15 lines of code. Use it to filter which clients receive each message.
       - On client disconnect: for each topic in client's Set, call app.mqttService.unsubscribe(topic). Remove client from map.
